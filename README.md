# Open Line

A config-driven civic-engagement PWA platform. It turns news about an elected official into a
phone call to their office: read a story, get grounded talking points, tap to dial.

**One codebase, one `config.json` per politician.** Adding a new official is a config change, not
a rewrite. Launch instances:

- **CallDoug** — Premier Doug Ford (Ontario, provincial)
- **CallCarney** — Prime Minister Mark Carney (Canada, federal)
- **CallChow** — Mayor Olivia Chow (Toronto, municipal)

> Naming convention: uniform **Call(NAME)**, matching each politician's own public brand for
> first-vs-last name (Call **Doug**, Call **Carney**, Call **Chow**). Wordmark is the closed
> compound that matches the domain (`calldoug.ca`, `callcarney.ca`, `callchow.ca`).

## Editorial stance — uniform accountability journalism

Every officeholder gets the **same** critical accountability lens, regardless of party. This is
not a fan site and not an opposition campaign. The stance is baked into the shared prompt
(`pipeline/prompts/summarize-and-extract.txt`) and disclosed on each instance's About page.

## Stack

Vanilla HTML/CSS/JS + [Vite](https://vitejs.dev) + pnpm. Static JSON per instance, no backend.
Hash-routed PWA with a per-instance service worker and manifest. localStorage for Saved/Called.

## Develop

```bash
pnpm install
pnpm dev:calldoug      # http://localhost:5173  (or: INSTANCE=callcarney pnpm dev)
```

The `INSTANCE` env var selects which folder in `instances/` parameterizes the build. The dev
server and the production build serve the same per-instance `config.json` / `articles.json`.

## Build (all instances)

```bash
pnpm build             # -> dist/calldoug/, dist/callcarney/, dist/callchow/
```

Each `dist/<id>/` is a self-contained static site (relative asset paths), ready to deploy to its
own domain/subpath: `calldoug.ca -> dist/calldoug`, `callcarney.ca -> dist/callcarney`.

## Deploy (GitHub Pages)

`.github/workflows/deploy.yml` builds every instance, generates a landing page
(`scripts/build-landing.mjs`), and publishes one Pages site on every push to `main` (including the
daily feed-refresh commit) and on demand:

```
/            → issue-first meta-site: pick an issue (housing, transit…) and see it across all
               three levels of government at once; links into each instance's article
/calldoug/   → CallDoug      /callcarney/ → CallCarney      /callchow/ → CallChow
```

The root is built by `scripts/build-meta.mjs`, which aggregates every instance's feed into
`dist/issues.json` and renders an interactive browser. Cross-cutting issues (covered by ≥2 instances)
are marked ◆. Each instance also links back to it ("Browse one issue across every level →" in the
footer, via `platform.home_url`).

## Analytics (optional, privacy-respecting)

Off by default. To measure the platform's whole point — **`tel:` taps per week** (spec §9) — set
`analytics` in `platform.json`:

```json
"analytics": { "enabled": true, "provider": "plausible", "src": "https://plausible.io/js/script.js", "domain": "openline.ca" }
```

Per-instance override: add `"analytics_domain": "calldoug.ca"` to an instance `config.json`. When
enabled + a domain is set, the build injects [Plausible](https://plausible.io) (no cookies, no personal
data — GDPR-friendly). Custom events fired: **Call: phone**, **Call: email**, **View call screen**,
**Save**, **Mark called**, **Share** (each tagged with the instance). When enabled, the About page
shows a short privacy note. Umami works too — just point `src`/`domain` at your instance.

Enable it once: repo **Settings → Pages → Source: GitHub Actions**. No secrets needed.

**Custom domains:** a single Pages site supports one custom domain, so the natural setup is one
domain at the root (e.g. `openline.ca`, with instances at `openline.ca/calldoug`). To put each
instance on its **own apex domain** (`calldoug.ca`, …) you need either one repo per instance or a
host that supports multiple projects (Cloudflare Pages / Vercel — the latter is already in the
sibling `pantry`/Tic-Tac-Done stacks). The build output is per-instance, so either path works
without code changes; only the cross-instance footer URLs in `platform.json` would point at
subpaths vs. apex domains.

## Repo layout

```
src/            Shared PWA. NEVER edited to add an instance. Carries zero politician-specific data.
instances/<id>/ config.json + articles.json (articles.json is generated by the pipeline).
platform.json   Platform name + the cross-instance roster shown in every footer.
pipeline/       RSS fetch + Anthropic summarize (scaffolded; CI wiring is Phase 2).
vite.config.js  Reads INSTANCE, injects config, emits per-instance assets into dist/<id>/.
scripts/        build-all.mjs loops every instance.
```

## Add a new instance (target: under an hour)

1. `mkdir instances/<id>` and write `config.json` (copy an existing one; fill in name, title,
   jurisdiction, **verified** office contacts, RSS sources, filter keywords, topic tags,
   accent color, tagline).
2. Add `articles.json` (the pipeline generates it; for a manual seed, follow the schema in an
   existing instance — see the §4.5 shape: `id, title, source_name, source_url, published_at,
   summary, tags[], talking_points[{point, source_reference}], suggested_ask, tone_note`).
3. Add the instance to the `siblings` list in `platform.json` so the cross-instance footer links it.
4. `INSTANCE=<id> pnpm dev` to preview, `pnpm build` to produce `dist/<id>/`.
5. Point a domain at `dist/<id>/`.

**No edits to `src/` are required.** If you find yourself touching shared code to add an instance,
that's a bug in the abstraction — fix it there, not in the instance.

## Guardrails (non-negotiable)

1. Never fabricate — every claim traces to the linked source article.
2. The source link is always the top action on an article page.
3. Talking points are prompts, not scripts ("say it in your own words").
4. Respectful contact only — staff answer these phones.
5. No personal info beyond public office contacts.
6. AI involvement is disclosed on every article.
7. Corrections are welcome (link on every article).

## Pipeline (automated, daily)

The pipeline refreshes every instance's `articles.json` from live RSS:

- `pipeline/fetch-articles.mjs` — pulls `config.rss_sources`, keyword-filters, dedupes against the
  existing feed, and grabs a best-effort article body (feed `content:encoded`, else page extraction).
- `pipeline/summarize.mjs` — generates the summary + talking points via **GitHub Models** (free LLM
  inference) using JSON-schema **structured outputs** (tags enum-locked to that instance's `topic_tags`,
  schema-valid by construction). The provider lives in one `callModel()` function so it can be swapped
  later (see Roadmap). Prepends results, trims to 200, rolls overflow into `instances/<id>/archive/YYYY-MM.json`.
- `pipeline/update.mjs` — the entrypoint; `--all` auto-discovers every instance (no per-instance code).

**Cross-level stories.** Each instance declares a `level` (`federal`/`provincial`/`municipal`), and
the summarizer reports which levels a story implicates. After every instance summarizes its own
keyword matches, `update.mjs` runs a cross-post pass: a story flagged as multi-level is *offered* to
the other instances at those levels, and each instance's own prompt validates relevance and writes
its own framing. So one federal–provincial housing deal that cities must deliver lands on **CallDoug,
CallCarney, and CallChow** — three different accountability summaries of the same event. The article
page shows a "spans federal · provincial · municipal" banner linking the sibling instances.

**Free + autonomous — no paid API key.** Summarization runs on GitHub Models, authenticated with the
GitHub token. In Actions that's the built-in `GITHUB_TOKEN` (granted `models: read` in the workflow) —
**nothing to create or store, no bill**. Locally, export a PAT with the "Models" read permission as
`GITHUB_TOKEN`.

```bash
pnpm update:dry           # fetch + filter + dedupe only — no model calls, no writes (safe anytime)
GITHUB_TOKEN=… pnpm update          # live: summarize new articles and write feeds
node pipeline/update.mjs calldoug   # one instance
```

**Scheduling:** `.github/workflows/update-feeds.yml` runs `update.mjs --all` **daily** (11:00 UTC;
change the cron to `0 */4 * * *` for every 4 hours), commits any changed feeds, and pushes. It needs
**no secrets** — the built-in token covers both committing and model inference. Adding an instance
needs no workflow change.

> Feed URLs and office phone numbers drift — verify them before a deploy (spec risk register). The
> pipeline tolerates a dead feed (logs it and continues), so one broken source won't stop a run.

## Roadmap

- **Pluggable summarizer** — the LLM call is isolated in `callModel()` in `summarize.mjs`. A future
  version should make the provider configurable and **A/B compare summary quality across GitHub Models
  (current), the Anthropic API, and free tiers of Google Gemini / Groq** — then pick per-instance or
  by cost/quality. (Switching providers is a one-function change plus a token.)
- Per-apex-domain hosting (`calldoug.ca`, …) — needs one repo per instance or Cloudflare/Vercel
  (current deploy is one Pages site with subpaths + a landing page).
- Platform meta-site at `openline.ca` (the issue browser is the seed of this).
- Community features (spec §8 Phase 5): story suggestions, collective call counter, weekly digest.
